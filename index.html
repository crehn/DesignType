How Do You Design Software?

Find out by filling out this questionaire.

Introduction

As developers we frequently make design decisions. They can be as general as "How many levels of abstractions do I need?" or "Which framework should I use?" and as concrete as "How to name this method?" and "In which order should these values be supplied?". They center around modularization (how to break a problem down into classes and methods), interface design (how to craft these modules such that they are easily useable) and internal module structure (how to compose the inner workings of classes).

Unfortunately there is no single correct solution for any of these questions. There are many bad solutions but there typically are several "good" ones. Deciding among these good solutions is not an objective task as each of them has different benefits and drawbacks. So the decisions are made in a subjective way which depends on a developer's way of thinking.

The different attitudes may lead to frequent arguments on which way is best. The following questionaire may help to find out how you make design decisions. You may even try to use it to find out how your collegues think about software design. We hope that a better understanding of how other developers think, improves the communication in a team.

Usage: In each group check exactly fife statements. Use those you can identify with most, those statements you could make yourself.

Simple vs. Powerful

Simple
  • "A simple solution is always better than a complex one."
  • "A class is good when it's small. Large classes are a smell."
  • "A method should not be longer than ten lines of code."
  • "If you don't need it (yet), don't implement it (yet)."
  • "Don't provide configurability if you don't need to."
  • "Avoid generics, polymorphism, reflection, and other complicated programming language features."
  • "Avoid low-level optimization of algorithms especially when involving assembler, bit-operations, and pointers. Slower implementations will work just fine."
  • "Use simple brute-force solutions instead of complicated algorithms. Slower algorithms will work in the first place."
Powerful
  • "A solution is always better when it solves more problems."
  • "Code is beautiful when it solves many problems at the same time."
  • "If there is a library or framework, use it. Don't reinvent the wheel."
  • "If you build something, think about non-functional requirements like performance, scalability, changability, portability, etc. early on."
  • "A configurable solution is better than a hard-wired one."
  • "I like generics, exceptions, polymorphism, closures, operator overloading, aspect-orientation, reflection, annotations, etc."
  • "I like code generators, configurable builds, and SOAP."
  • "If you get a lot more functionality with a little bit of extra work, don't think for too long. Do it."

Abstract vs. Concrete

Abstract
  • "I think in terms of abstractions, concepts, and models"
  • "In most cases duplication can be eliminated by finding the right abstractions."
  • "Each class should represent a real-world concept."
  • "Birthday is a perfectly valid class. It's more than a pure Date as it can, for example, compute an age."
  • "CustomerNumber should be a class in order to facilitate type checking."
  • "Class names which end with Object, Manager, Handler, Data, Info, etc. are a smell."
  • "Each class and each method should be understandable on it's own, i.e. without knowing the rest of the system. If that's not the case, a suitable abstraction is missing."
  • "An interface shouldn't expose the inner workings of a module."
Concrete
  • "I think in code (i.e. in terms of classes methods and statements)"
  • "Duplication is the root of all evil. Avoid it at any cost."
  • "I frequently extract classes when I see that this improves the structure of the code."
  • "Avoid tiny classes which don't provide functionality."
  • "Coding is the repetions of writing code and extracting redundancy."
  • "It is more important for the implementation to be simple than the interface."
  • "You can read good code from top to bottom. And this is a good way of understanding it."
  • "You can refactor code without completely understanding it."

Pragmatic vs. Idealistic

Pragamtic
  • "Time to market is sometimes more important than code quality."
  • "I'm not afraid of duplicatinmg code... if there is a good reason to do so."
  • "Often you get 80% of the solution for 20% the effort. We can care about the remaining 20% manually or partly automated if needed afterwards."
  • "I only implement what's necessary. I don't care about completeness and symmetry. If it's not needed it's worthless."
  • "Surprisingly many badly written systems in practice work quite well for quite a long while. There is no need to produce art."
  • "It's the result that counts. Not the tiny details. A workaround is better than no solution at all."
  • "Don't care about non-functional requirements which are not relevant for the system. If the system will never be ported, it doesn't need to be portable."
  • "I don't have to understand all the code I find at stackoverflow. If it works, it's good."
Idealistic
  • "Bad code quality will kill you in the long run."
  • "Avoid duplication. Period."
  • "Classes are meant to represent concepts. If they don't, the code will be less understandable."
  • "Code is good, when it's symmetric. Classes should have all expected methods. Don't implement a class for complex numbers which can add but not subtract."
  • "Don't misuse programming language constructs. Don't missuse frameworks, patterns and concepts. Don't say REST when it's actually not."
  • "Each piece of functionality has it's place where it belongs to. It should be implemented exactly there."
  • "If in doubt, invest two more minutes for finding a good identifier. If there is no good identifier, that's a sign that the abstraction is wrong."
  • "Each module (package, class, method) and even each line of code should be understandable on it's own, i.e. without knowing the rest of the system."

Technology vs. Stability

Technology-Oriented
  • "Technology evolves. We should, too."
  • "I prefer keeping up with the times by frequently updating to new technology."
  • "Move fast and don't fear breaking things."
  • "New technology often is an improvement. We should enbrace that in order to get better."
  • "Upgrading technology regularly is easy as it's always a small step. Waiting for too long creates huge unnecessary efforts."
  • "I like trying out new frameworks, libraries, design strategies, and languages. Even if I won't use it in practice, it's good to look left and right."
  • "Stagnation kills creativity."
  • "We build software such that it's easily fixed when there is a problem. Roll forward is typically better than roll back."
Stability-Oriented
  • "If it works, there is no need to change it."
  • "I don't fear being left behind by evolving technology. Let's first see what comes out on top."
  • "I like stable technology which stood the test of time."
  • "New technology often is just a hype. It won't last long and the next one will be praised as the solution to all problems."
  • "There is no silver bullet. That's an often neglected fact."
  • "Better double check."
  • "Too much creativity is risky when you already have accomplished something. You need to find the correct amout of creativioty which fits the business."
  • "Prepare to rollback. If something goes wrong, we can still switch to a working solution."

SCPS: MacGyver or The Fire Fighter

Attributes
Simple, Concrete, Pragmatic, Stability-oriented
Description
You can fix anything with a string of duct tape. When there is a problem, you are the one to find a simple and efficient solution for it. You don't like endless discussions without getting anything done. While others may still be dreaming of a big system which can do anything, while they spent large amouts of time keeping up with the latest hype, you have already solved the problem and begin looking around where the next problem can be fixed. You don't know everything but when there is something you cannot solve yourself, you know where to find helpful code snippets in the internet.
Your designs are
Small, simple and efficient but probably less foresightful
Programming is
Problem solving
Principles you probably like
KISS, MIMC, RoE
Principles you rather disregard
GP, MP, PSU
Strengths
  • You are an efficient problem solver
  • You don't loose time in endless discussions
  • You simply get things done
Suggestions
  • Try not to loose the big picture
  • Your solutions work now pretty well but in the future they may need to be revised. Be careful that all those necessary future changes don't add up such that the overall effort is higher than that for the more elaborate solutions of you collegues.
  • Don't forget to refactor. Otherwise all those simple enhancements may result in something complicated.

  • Worse is better

Final Remarks

This categorization is by far not covering all aspects developers have. Working style, prefered development process, people skills, and many other aspects are neglected here. That's also not the goal of this questionaire. Rather the hope is that this structure helps you to understand how your collegues think when they make design decisions.

This questionaire is still work in progress. It still hasn't been tried out how well the mapping of the selected statements to design types works. Neither it's clear whether the design types are described correctly. Freedback welcome.
